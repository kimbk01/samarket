# SAMarket Cleanup Policy

이 문서는 개발 중 삭제하면 위험한 파일과, 개발 종료 시점에 삭제할 파일을 안전하게 관리하기 위한 기준이다.

## 분류

- `production`
  현재 운영 기능 또는 운영 전환 예정 핵심 코드. 삭제 금지.
- `dev-only`
  로컬 개발 보조용 코드. production 에서는 노출되면 안 된다.
- `test-only`
  QA, 테스트 계정, 테스트 라우트용 코드. 기능 플래그 뒤에서만 유지한다.
- `mock-backed`
  실제 기능을 지탱하지만 데이터 소스가 mock/sample 인 코드. 대체 API/DB 준비 전까지 삭제 금지.
- `deprecated`
  대체 구현이 이미 있고, 내부 참조 제거가 끝나면 삭제할 코드.

## 삭제 기준

아래 조건을 모두 만족할 때만 삭제한다.

1. 정적 import 참조가 없다.
2. 라우트 진입점으로 직접 접근되지 않는다.
3. 동적 import 또는 feature flag 연결이 없다.
4. 운영/QA 플로우에서 현재 사용하지 않는다.
5. 대체 구현이 이미 존재한다.
6. 최근 작업 중인 변경과 충돌하지 않는다.

## 운영 규칙

- `mock-*`, `test-*`, `demo-*`, `dev-sample-*` 파일은 바로 삭제하지 않는다.
- 이런 파일은 반드시 `deploy-surface` 또는 전용 정책 함수로 노출을 제어한다.
- 관리자용 샘플/통계 목 데이터는 실제 API 전환 전까지 `mock-backed` 로 유지한다.
- `@deprecated` 파일은 새 코드에서 import 하지 않는다.

## 종료 시점 정리 순서

1. `test-only` 표면 제거
2. `dev-only` 표면 제거
3. `mock-backed` 대체 완료 확인
4. `deprecated` 파일 삭제
5. 생성물(`.next`, `coverage`, `build`, `out`) 정리

## 현재 프로젝트 원칙

- 개발 중에는 분류와 노출 제어를 먼저 한다.
- 삭제는 “기능 대체 완료 + 참조 제거 확인” 이후에만 한다.
- 실서비스 전환 직전에는 이 문서와 `docs/cleanup-inventory.md` 를 기준으로 최종 삭제 배치를 수행한다.
